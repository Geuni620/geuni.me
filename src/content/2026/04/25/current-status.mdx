### 화면개발

프론트개발자로서, 화면개발은 다음과 같이 변화되었다.

Figma에서 디자이너가 디자인을 완료해주면 개발자모드로 화면에 치수를 확인하고 개발을 진행했었다.

하지만 최근에는 `Figma MCP`를 통해 적당한 범위의 node-id를 Focus하고, 해당 URL을 claude-code에게 붙여넣는다.

그리고 하위 node-id를 탐색해서 디자인 화면을 잡아달라고 요청한다.  
컨텍스트 맥락을 여러번 주입해야하는 경우는 Claude.md나 Agents.md파일에 기록된 rules에 기록해둔다.

기존에는 로컬서버를 띄워서 런타임에서 새로고침하며 화면을 잡는 걸 선호했던 반면, 최근에는 `Storybook`에서 화면을 먼저 잡아보곤한다.

<SideNote>
  나 뿐만 아니라 다른사람도 느끼고 있는 것 같다. 심심찮게 인간이 병목이라는 말을
  많이 듣고 보곤한다.
</SideNote>
AI가 나보다 훨씬 빠른 타자속도를 가지고 있어서 순식간에 파일체인지 몇 백, 많게는
천 줄이상 발생하는데, 이걸 따라가는게 병목이었다.

차라리 이럴거라면 내가 입력하고 말지하는 생각이 들더라.

그래서 Storybook에서 화면 디자인을 먼저 어느정도 잡아놓고, 눈으로 확인하며 어긋난 부분을 체크한 뒤 Figma MCP를 통해 더 하위의 node-id를 전달한다.

### 서버 API

이전 회사에서 Swagger 기반으로 Mock up 데이터와 Api Call, Service Layer 등을 `Orval`로 세팅해서 사용하곤 했었다.  
현 회사에 왔을 땐, Server State를 정의하는 컨벤션이 존재했고, [Orval](https://orval.dev/)이나, [Hey-api](https://heyapi.dev/) 등등 기존의 code-generator로 이를 강제할 수 없었다.

만약 code-generator의 사용으로 바꿔야한다면 전체적인 컨벤션 자체를 모두 변경해야하는 큰 작업이 될텐데, 팀원을 설득할 자신도 없었고, 설령 설득해도 잘 될거라는 보장이 없어서 그냥 기존의 방법으로 진행하곤 했다.
여기서 말하는 기존의 방법은 다음과 같다.

- 서버에서 Swagger를 업데이트 해준다.
- Swagger JSON을 소스코드 내 server-state 컨벤션에 맞게 파일 하나씩 생성하면서 만듦
- shared파일 내에서 사용되고 있는 enum 타입이 있다면 해당 enum 타입으로 타입 자체의 범위를 좁힘
- MSW로 Storybook에 띄워서 잘 노출되는지 체크해봄

이 과정에서 만약 초기에 전달해준 Swagger의 변경사항이 생겼다면, 서버개발자에게 어떤 부분이 변경되었는지 확인하고 Swagger를 다시 확인해본 뒤 변경점을 찾아서 일일이 고쳤다.
개인적으로 이 과정이 너무 불필요하고 귀찮았다. 그래서 서버개발자들이 API 수정이 발생하는 걸 극도로 싫어했다.

특히 단순 Field명의 변경은 극도로 대응하기 싫었다.
requestId와 requestNo와 같은 변경을 대응하는게.. 너무 귀찮고 현타가 왔다.

<br />

전사에서 claude-code를 지원해준 뒤 가장 먼저 만든건, Swagger기반 Code Generator였다.  
Hey API나 Orval이 아닌 현재 우리가 사용하고 있는 컨벤션에 맞춤형 Swagger Code Generator였다.

물론 Swagger URL을 전달하고 curl로 요청보낸 뒤 응답받은 JSON파일 기반으로 특정 endpoing를 지정해서 요청하며 컨벤션에 만들어달라고 해도 잘 만들어줬다.
하지만 이 모든건 context이며, 한 두번 요청하면 컨텍스트가 눈에 띄게 차오르는 것을 확인할 수 있었다..

또한 결정적인 작업이 아닌 비결정적이었다. 즉, 10번의 동일한 프롬프트로 claude-code에게 요청을 해도, 10번 다 동일한 결과를 보장받을 수 없었다.
결정적이게 만드는 방법은 script라고 생각해서, Swagger 기반으로 대략적인 초안용 code-generator를 만들었다.

<br />

[clack](https://github.com/bombshell-dev/clack)로 대화형으로 터미널 내에서 필요한 요소들을 체크해서 완성하면 script기반으로 code-generator가 컨벤션에 맞게 생성되도록 했고,
초안이다보니 tanstack-query queryKey 같은건 주석으로 @todo, @fixme 같은 걸 남겨서 내가 확인하며 작업했다.

또는 claude-code에게 현재의 소스코드 변경점을 기반으로 컨벤션 구조에 어긋나는게 있다거나, 누락된게 있다면 조정을 부탁했을 때 대체로 매끄럽게 완성되곤 했다.
해당 작업을 버전 1이라고 했을 때, 1~2개월 정도는 잘 사용하다가 불편한 것들이 생겨서 다음의 사항을 추가적으로 수정했다.

- 대화형의 깊이가 너무 깊어질 경우, 예를들어 `apps 선택 > controller 선택 > method + endpoint 각각 선택`의 과정은 너무 깊이가 깊었다.
- create는 만들었는데, update를 누락해서 한번 만들어놓은 요소를 수정할 때면 큰 파일 체인지가 발생했고 초기에 예상했던 swagger의 field 변경점만 체크하기 어려웠다.
- 생각보다 내가 직접 핸들링하는 경우가 점점 줄어들었다. 위에서 언급했듯 더이상 코드를 내가 작성하지 않았다. claude-code에게 위임하는 경우가 많았다.

위 불편사항을 다음과 같이 개선했다.

- create만 만들었었는데, update도 만들었다.
- 더이상 터미널에 script를 돌리지 않는다. 대신 claude-code에게 script 명령어를 알려주고 특정 endpoint를 요청하면 알아서 컨벤션에 맞게 구성해준다.
- 서버개발자도 더이상 일일이 변경점을 구두로 전달해주시지 않는다. 대신 변경점이 발생한다면 슬랙 하위에 변경점에 대한 md파일을 첨부해서 전달해주신다. 이는 skills로 만들어서 첨부해주시는 것 같다.
  - 나는 slack 메시지를 복사해서 claude-code에게 붙여넣고 해당 작업을 지시한다. 그리고 맥락을 같이 읽어보며 궁금한게 있으면 claude-code에게 물어보고, 확인이 필요한 사항은 서버개발자에게 확인한다.

### 테스트

사실 테스트는 크게 변한게 없는 것 같다.  
개발이 완료되고 배포를 하기 전에 하나씩 눌러보면서 확인하고 라벨의 변경이 잘 되었는지, 디자인이 깨지지 않았는지, 기능이 온전히 작동하는지는 사람이 하나씩 눌러본다.  
또한 페이지 로드가 느리지 않는지, 이미지는 잘 뜨는지 등을 확인해보곤 한다.

앞으로는 어떻게 될지 예상할 수 없지만 이런 디테일한 영역은 AI가 쉽게 대체하지 못할 것이라고 생각한다.  
AI가 이미지 크기가 너무 커서 로드가 늦다거나, 레이아웃 시프트가 발생해서 사용성이 좋지 못한 것들 등등  
사람이 직접 사용해보면서 경험적으로 체득한 부분들은 설명하기 조차 어렵다.

<br />

가장 최근에 프로덕트를 리뉴얼하면서 디자인 시스템의 Token만 사용하고 그 외의 요소는 디자이너가 잡아준 UI를 기반으로 작업했었다.
즉, 디자인 시스템을 사용했다면 잡혀져있을 요소이지만 Figma mcp를 사용해서 작업하다보니 누락된게 엄청 많이 발생했다.

감사하게도 디자이너가 QA를 꼼꼼하게 진행해줘서 Task들이 많이 쌓였는데, 하나씩 Agents에게 전달하면서 수정하는게 너무 힘들고 번거롭더라, 생각보다 시간도 오래걸렸다.
그래서 codex로 해당 작업을 위한 하네스를 구축했다. 해당 부분까지 작성하면 글이 너무 길어질테니.. 이건 따로 하네스 구축이 완료되면 다시 글을 남길 예정이다.

## 업무 흐름을 UI에서 CLI로 옮겼다.

우리 회사는 현재 Jira를 사용한다.  
업무를 시작하기 전에 story티켓 하위에 subTask를 딴 뒤, 생성된 subTask의 이슈번호로 브랜치를 생성해서 작업을 시작한다.

subTask를 생성하기 위해 Jira에 들어가서 Task 생성을 누르고, 담당자를 배정하고, 제목을 작성하고.. 더이상 이런 작업을 하나씩 입력하고 누르지 않는다.  
터미널에서 `task-init`을 누른 뒤 제목/담당자/부모티켓을 입력하면 master를 기준으로 브랜치를 생성해주고, jira에 subTask 티켓도 따준다.

필요한 브랜치의 작업을 이어가려면 해당 브랜치를 rebase해서 최신화 시킨 뒤 작업을 진행하면 된다.

즉, 터미널을 기준으로 화면을 벗어나는 일이 크게 줄었다.

QA 작업 이후에 생성된 티켓도 마찬가지이다. design-qa와 같은 스킬과 함께 링크를 첨부하면, 하위에 subTask들을 모두 읽은 뒤, subTask의 이슈사항과 매칭되는 figma 링크, 수정해야할 소스코드의 변경점을 체크해서 내게 테이블로 보여준다.
나는 하나씩 링크를 클릭하고, 변경점이 맞다면 작업을 implement 단계로 넘기고, 그렇지 않다면 figma 링크를 찾아서 해당 링크로 업데이트해달라고 요청한다. 그리고 implement로 작업을 넘긴다.

<br />

또한 작업이 완료되면 create-commit, create-pr 올려놓는다.
